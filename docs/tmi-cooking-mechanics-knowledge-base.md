# 东方夜雀食堂料理机制知识库（评分与标签）

更新日期：2026-05-06
适用范围：本仓库后续算法设计、解释文档、规则校验脚本

## 1. 目标与边界

本知识库仅保留可直接用于实现的机制内容，不包含来源说明、证据分级和交叉验证流程。

覆盖内容：

- 稀客评分机制
- 标签生成、相消与禁忌机制
- 特殊顾客的改判与特殊规则（重点含古明地恋）
- 本仓库内部命名与伴随窗口展示规范

## 2. 本仓库内部命名与展示规范

### 2.1 评分命名替换

为避免“粉-橙-绿-紫-黑”颜色词在文案与实现中产生歧义，本仓库统一使用：

- 完美（ExGood）
- 满意（Good）
- 普通（Normal）
- 不满（Bad）
- 极差（ExBad）

旧称呼与新命名对应关系：

- 粉 -> ExGood
- 橙 -> Good
- 绿 -> Normal
- 紫 -> Bad
- 黑 -> ExBad

### 2.2 展示色彩规范

伴随窗口标签底色保持颜色语义，但内部逻辑名一律使用上面的五级命名。

建议标准色：

- ExGood：粉色，`#EC4899`
- Good：橙色，`#F59E0B`
- Normal：绿色，`#22C55E`
- Bad：紫色，`#A855F7`
- ExBad：黑色，`#111827`

### 2.3 相消标签颜色规范

所有“被相消/被覆盖”的标签，统一使用棕色作为标准色。

建议标准色：

- Cancelled Tag（相消标签）：棕色，`#8B5E3C`

建议展示方式：

- 棕色底或棕色文字
- 同时加删除线，表示该标签不参与最终有效标签判定

## 3. 稀客评分机制（实现口径）

### 3.1 点单结构

每次稀客点单包含两项要求：

- 一项料理标签要求
- 一项酒水标签要求

### 3.2 基础计分

- 满足一个顾客喜好标签：`+1`
- 命中一个顾客厌恶标签：`-1`
- 料理与酒水都参与总分计算

### 3.3 五级评价判定（仓库标准）

按以下顺序判定（从上到下命中即停止）：

1. ExGood：总分 `>= 4` 且同时满足料理要求与酒水要求
2. Good：总分 `>= 3` 且至少满足一项点单要求
3. Normal：总分 `>= 2` 且至少满足一项点单要求
4. Bad：总分 `>= 1` 但未达到上述条件
5. ExBad：总分 `<= 0`

说明：

- 该命名用于本仓库逻辑与文档。
- 展示层仅做颜色映射，不使用颜色词作为逻辑字段名。

### 3.4 默认酒水假设与加料上限（节省食材）

由于无法预先确定玩家最终采用的“菜品 + 酒水”组合，本仓库在稀客料理推荐中采用如下默认假设：

- 默认酒水已满足点单酒水要求
- 默认酒水贡献固定 `1` 分

据此，料理侧仅追求 `3` 分即可达到 ExGood 所需总分，不再为料理分数超过 `3` 继续加料。

实现约束：

- 料理搜索优先目标：在当前“最多加料”限制内尽量达到 `foodScore >= 3`；达到后停止继续加料，但最终 `foodScore` 与评级仍按实际有效标签分计算（不做封顶）
- 若当前解锁内容或库存无法达到 `foodScore >= 3`，仍必须返回满足点单料理 tag 的最高分候选；只有完全没有满足点单 tag 的候选时才显示无推荐
- 在达到目标分前提下，优先更少加料数量
- 当多个食材可提供同一目标标签时，优先选择持有数更高者；持有数相同则优先价格更低者
- 展示排序优先降低经营操作复杂度与资源压力：同分时加料种类少者靠前，其次低库存压力更小者靠前，再按料理售价、加料成本和料理 ID 兜底
- 加料展示标注用途标签：`+食材(标签1,标签2)`

该策略的目的：

- 降低“为无效超分追加食材”带来的浪费
- 提供更稳健、可复现的推荐结果

## 4. 标签机制

### 4.1 标签来源

最终料理标签由三部分合并：

- 料理自带标签
- 加料带入标签
- 动态标签（价格、总食材数、流行）

### 4.2 动态标签

- 菜价 `< 20`：追加“实惠”
- 菜价 `> 60`：追加“昂贵”
- 总食材数 `>= 5`：追加“大份”
- 命中当期流行喜爱：追加“流行喜爱”
- 命中当期流行厌恶：追加“流行厌恶”

实现口径补充：

- 先计算静态动态标签（实惠/昂贵/大份）与明星店效果，再做互斥覆盖。
- 料理流行喜爱/厌恶标签基于“互斥后的有效标签”判定，避免出现“被覆盖标签仍触发流行效果”的连锁残留。

### 4.3 标签相消（覆盖）

当前按有向覆盖处理，强标签覆盖弱标签，弱标签被标记为相消（棕色 + 删除线）：

- 肉 覆盖 素
- 大份 覆盖 小巧
- 饱腹 覆盖 下酒
- 重油 覆盖 清淡
- 灼热 覆盖 凉爽

### 4.4 禁忌与黑暗物质

存在禁忌标签组合。命中禁忌时应视为无效推荐组合并在推荐阶段剔除，避免进入可选结果。

补充说明：

- 黑暗物质不作为常规料理推荐候选。
- 梅蒂欣角色卡说明统一使用固定句式：梅蒂欣对黑暗物质的评价为3分。
- 该说明仅用于提示彩蛋结论；推荐层仍不并入黑暗物质常规候选。

## 5. 特殊顾客机制（重点补充）

### 5.1 彩蛋规则优先级

- 对“满意或以上”的彩蛋料理，推荐阶段提供对应的得分下限（score floor），但不跳过点单料理 tag 约束。
- 若未满足点单料理 tag，则不会被改判为 ExGood。
- 列表排序按分组处理：星标+彩蛋 > 星标 > 彩蛋 > 其他（彩蛋分组需满足点单料理 tag）。
- 彩蛋“特殊显示”由设置项 `rareEasterVisualEnabled` 控制，默认关闭；关闭时仅保留推荐计算，不显示彩蛋说明、彩虹样式与彩蛋置顶。
- 彩蛋说明统一显示在稀客角色卡片区域，不在料理卡片区域重复展示。
- 正面彩蛋可独立控制“是否置顶”和“是否高亮料理边框”。

### 5.2 当前接入的彩蛋规则

满意或以上（优先展示）：

（注：当前为“加分下限”机制，不再无条件置顶展示）

- 蕾米莉亚 × 猩红恶魔蛋糕
- 饕餮尤魔 × 油豆腐
- 雾雨魔理沙 × 牛肉鸳鸯火锅
- 河城荷取 × 黄瓜（食材级，仅高亮加料食材边框，不高亮料理卡片边框，也不参与彩蛋置顶）

不满或更低（直接禁推）：

- 古明地恋 × 无意识妖怪慕斯
- 绵月丰姬/绵月依姬 × 蜜桃红烧肉
- 犬走椛 × 可可豆（食材级）

实现约束：

- 禁推规则优先级高于普通评分与排序。
- 彩蛋优先展示仍遵循“用户可用内容”边界：未拥有、被过滤或不可用的料理不会被强行展示。

### 5.3 彩蛋说明文案规范（用户侧）

- 彩蛋说明仅允许使用固定句式：`{角色}对{料理或食材}的评价为{评价内容}。`
- 禁止追加任何推荐机制说明、排序说明、点单命中说明或实现细节。

当前已接入文案（与页面保持一致）：

- 蕾米莉亚对猩红恶魔蛋糕的评价为4分。
- 饕餮尤魔对油豆腐的评价为3分。
- 雾雨魔理沙对牛肉鸳鸯火锅的评价为4分。
- 河城荷取对黄瓜的评价为3分。
- 古明地恋对无意识妖怪慕斯的评价为极差。
- 绵月丰姬对蜜桃红烧肉的评价为极差。
- 绵月依姬对蜜桃红烧肉的评价为极差。
- 犬走椛对可可豆的评价为极差。

## 6. 运行时映射约束

### 6.1 料理流行标签

- Mod 优先从当前游戏运行时读取流行喜爱、流行厌恶和明星店状态；不读取 `.memory` 文件。
- 当前仓库允许“流行喜爱”和“流行厌恶”同时存在，分别作为独立偏好字段处理。
- 游戏运行时中的料理流行值可能保存为数字 tag id；该映射已通过游戏资源 `FoodTagProfile.asset` 与中文 `FoodTagsLang.txt` 核实。
- 已确认示例：`7=清淡`、`19=招牌`、`-3=昂贵`、`-2=实惠`、`-1=大份`。
- 运行时检测到 `Aya_FamousIzakaya=true` 应视为“明星店效果已开启”，并写入推荐状态 `famousShopEnabled=true`。
- 当明星店效果开启且流行喜爱 tag 为 `招牌` 时，`招牌` 不写入 `popularFoodTag`，而是仅启用明星店效果，避免占用“流行喜爱”槽位。
- 当前只接纳推荐逻辑可直接消费的料理标签；厨具标签、`黑暗物质`、`厨具「夜雀」` 等系统型 tag 继续忽略。
- 若运行时数组里存在多个值，只取第一个能作为当前料理标签字符串识别的有效值；没有有效值则按未设置处理。
- 解析结果写入推荐状态 `popularFoodTag` 与 `popularHateFoodTag`，由普客与稀客推荐共同消费。
- 历史资料和社区资料只用于交叉佐证标签语义；实际数字枚举源以游戏本体资源提取结果为准。

### 6.2 暂不接入的运行时字段

- `popLikeBevTags`、`popHateBevTags` 当前只做解析保留，不驱动推荐。
- `rewindMode` 当前视为无效字段，不参与现有推荐与筛选逻辑。

### 6.3 联动稀客显隐

- 联动角色与 DLC9（MetaMystia 模组）角色是否可推荐，以当前运行时可用数据和出现数据为准。
- 伴随窗口不维护独立隐藏列表，也不通过额外配置让角色跨地区强行显示。
- 已实现映射：`MC_Gensokyo -> [30]`、`3FARIES_Collab -> [31]`、`TBS_Kokoro -> [41]`、`TBC2_Collab -> [36, 37, 38]`。
- 语义对应：`30=萌澄果`、`31=蹦蹦跳跳的三妖精`、`41=秦心`、`36/37/38=冴月麟/立空汐/时焉侑`。
- `THYG` 视为“已知空映射”的合法 key：允许出现，但不会增删任何角色。
- DLC9（MetaMystia 模组）角色在未检测到对应 DLC 激活时不作为默认推荐候选。
- 未知 `collabStatus` key 一律忽略，避免把未核证 key 错误映射到现有角色。

## 7. 本仓库落地约束（供后续开发直接使用）

- 逻辑层只使用 ExGood/Good/Normal/Bad/ExBad，不使用颜色词做枚举值。
- 展示层通过颜色映射渲染评级标签底色。
- 相消标签颜色固定为棕色，并必须可视化删除线。
- 推荐输出应同时保留：原始得分、改判后等级、触发的特殊规则名称，便于排错。
